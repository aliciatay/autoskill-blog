TL;DR

One paragraph

Chat with an AI does not compound. You solve a problem once — parse a messy CSV, hit an API, reshape a report — and next time you start from zero. AutoSkill is middleware that fixes this. You chat normally; in the background an Observer watches a rolling chat buffer (the last 40 messages), an Architect extracts parameterisable bits and generates a SKILL.md + Python script, validates it in a Docker sandbox, and an Integrator mounts the approved skill into the agent's workspace for every future session.

In one line: chat once, codify automatically, reuse forever.

What we built over the semester:

  • A three-phase pipeline — Observer → Architect → Integrator — behind a FastAPI backend with a Next.js 16 frontend.
  • A per-session Docker sandbox that runs the Gemini CLI agent and validates generated scripts before a user ever sees them.
  • Two-step skill generation (parameter extraction → parameterised skill), a layered validation ladder with auto-retry, and an embedding-based duplicate-skill check.
  • Beyond the proposal: skill evolution (refine an existing skill in natural language), Worker profiles, a session-scoped Files panel, and skill version history.

Why we built this

Non-technical users live inside chat interfaces. Chat is good for conversational problem-solving, but bad at compounding: every session starts cold, every workflow gets re-explained, and a prompt that worked last week has to be reconstructed from memory or a notes file the next time it is needed.

The industry's answer is Agent Skills: small, declarative packages that extend an agent's native capabilities. A skill is a directory with a SKILL.md spec, optional Python scripts, and optional references. Frameworks like the Gemini CLI pick them up automatically and route to them when a user's request matches the skill's description.

Skills are a good answer to the compounding problem. The catch: writing them is still a developer activity. You draft YAML frontmatter, articulate when the skill should fire, decide which values should be parameters, write the script, test it. That is a hard ask for someone who opened a chat tab because they did not want to write code.

Can a middleware layer passively observe chats, recognise the skill-shaped ones, and codify them automatically with high enough quality that a non-technical user only has to review and approve?

This report is the implementation write-up — the engineering decisions, tradeoffs, and close calls behind our particular version.

The shape of a skill

A skill is a folder that looks roughly like this.

csv_column_summary/ tree 
├── SKILL.md          # required — spec + frontmatter
├── scripts/
│   └── summarise.py  # optional — executable logic
├── references/       # optional — static docs the agent reads
└── assets/           # optional — templates, fixtures

SKILL.md's YAML frontmatter declares when the skill fires (a description the host uses for retrieval) and what inputs it takes. The Markdown body is prose instruction the agent reads when it decides to load the skill.

When the host agent sees a user message, it checks its available skills, matches on description, and — if a skill fits — loads the body and executes the referenced script with the extracted arguments. AutoSkill's job is to produce these folders automatically, from conversation history that was never meant to be a spec.

System overview

AutoSkill sits between the browser and the agent. The chat UX is unchanged, but every message also passes through an observation layer that keeps a rolling buffer, and a skill-synthesis pipeline runs asynchronously off that buffer.

Two details shape the rest of the system.

  1. Everything the agent executes runs inside a per-session Docker container. The chat agent is the Gemini CLI, spawned in our autoskill-sandbox image. Generated scripts are validated in the same image a separate sandbox runs later in production, so validation and execution share an environment.
  2. Storage is file-based, not a database. Skills, session state, and generated files all live on disk under the backend's working directory.

Observer — deciding when a conversation is a skill

The buffer

Every message is appended to an in-memory deque capped at 40 entries per session. We picked 40 because we wanted sufficient headroom for a trigger to still catch a workflow when the user drifts into unrelated follow-ups before deciding to save. The buffer is per-session, lives in a process-local dict, and is wiped on "New Chat" or after 2 minutes without a frontend heartbeat.

The trigger

After the assistant's reply finishes streaming, a lightweight Gemini Flash call — the trigger evaluator — reads the full buffer. The prompt frames the decision around one question:

After this conversation, who would perform the steps: the AI or the human?

An example. Both conversations look similar on the surface; only one is a skill.

Conversation A skill
User I have this sales.csv. Group by region and export totals as a chart.
AI [generates Python, runs it in the sandbox, pandas.read_csv → groupby('region').sum() → matplotlib.savefig('chart.png')]  "Done — here's the chart. The North region had the highest totals at $240K."
Conversation B not a skill
User How would I group data by region in pandas and make a chart?
AI "You can use df.groupby('region').sum() then plot.bar() to visualise. Here's a snippet you can run in your Jupyter notebook…"

Conversation A is a skill candidate. The AI produced an artifact (the chart) by executing a multi-step pipeline. Codifying this as csv_region_totals means: next time the user uploads products.csv and asks the same question, the same skill runs the same pipeline with different inputs. Same actor (AI), same actions, different data.

Conversation B is not. The AI produced text — instructions for the human to follow. Even if the instructions are perfect, there is no automatable workflow to extract: next time, the user still has to open a Python shell and type the code. Codifying this would just store the instructional text, which is already in the chat history.

The distinction: did the AI act, or did the AI teach? Only "act" conversations are automatable. Only automatable things become skills.

On top of that test, the prompt layers four filters. A turn only becomes a candidate if all four hold:

  • the output is concrete and executable — code, a file, a tool call result;
  • there is more than one distinct step;
  • the workflow is repeatable with different inputs;
  • the task completed successfully.

The full prompt is below. It is what the Gemini call receives after every assistant turn, with the rolling 40-message buffer appended as context.

backend/app/prompts/trigger.md prompt · verbatim excerpt 
You are an Observer agent monitoring a conversation between a user
and an AI assistant. Your job is to decide whether the conversation
contains a workflow that AutoSkill could package and re-execute
automatically as a reusable "Skill".

CORE TEST — ask yourself one question:
  "After this conversation, who would perform the steps:
   the AI or the human?"

  → AI performs the steps (generated code, a config, a document,
    a data output, executed a sequence of tool calls)
    = SKILL candidate.

  → Human performs the steps (the AI described instructions for
    the user to follow themselves) = NOT a skill.

A valid Skill candidate must also satisfy:
1. The output is concrete and executable (code, script, config,
   structured artifact) OR the assistant performed a sequence of
   automated actions
2. More than one distinct step was involved
3. The workflow is repeatable with different inputs (at least
   one parameterisable value)
4. The task completed successfully

CRITICAL AI FALLBACK CHECK:
Analyze HOW the AI solved the user's request using the
[System Alert] logs in the conversation:
- SCENARIO A: The AI strictly used an existing tool/skill and
  the outcome was perfectly successful without requiring custom
  manual code to finish the task. → NOT a skill candidate.
- SCENARIO B: The AI used an existing tool/skill, but it failed,
  so the AI had to manually write custom code/scripts to finish.
  → SKILL candidate (an evolution is needed).
- SCENARIO C: The AI wrote custom scripts/code from scratch
  without using any existing skills. → SKILL candidate.

Respond ONLY with valid JSON: {"detected": true, "name": ...,
"description": ...} or {"detected": false}.

Full buffer, not a rolling window

The proposal originally used a rolling window of the last N messages. We decided on a 40-message buffer so that skill-worthy workflows — which often straddle a clarification exchange (the user asks, the model asks back, the user answers, the model does the work) — aren't cut short before the parameterisable inputs appear.

Robustness

Three details made the trigger production-feasible:

retry
3-attempt cap with exponential backoff, but only on rate-limit and 503 signatures. Genuine failures fail fast.
timeout
60-second wall-clock wrapper via asyncio.wait_for; exceeded triggers are swallowed so the chat stream stays healthy.
non-block
We yield the {"done": true} event before awaiting the trigger result. The candidate arrives as a later SSE event when ready, or not at all.
dismiss map
Per-session _dismissed_candidates filters suppressed names so the same candidate is never re-surfaced until a genuinely new session.

Architect — from candidate to validated skill

If the trigger flags a candidate and the user clicks "Review Skill," the Architect runs.

Two-step generation

step 1
Parameter extraction. A Gemini call whose only job is to survey the conversation for hard-coded specifics (file paths, dates, column names, thresholds). Schema-constrained via Pydantic + response_schema. Temperature 0.2.
step 2
Skill generation. A second call that takes the extracted schema and produces the full SKILL.md, scripts, assets, and references as a SkillManifest. The prompt says "replace every example value from Step 1 with the corresponding {param} placeholder." Temperature 0.3.

This means that it is two API calls per skill — but it produces skills whose parameter schema and content are coherent with each other, because Step 2 is anchored on Step 1's output rather than inventing both in the same pass.

backend/app/prompts/param_extract.md step 1 prompt · excerpt 
You are analysing a conversation between a user and an AI assistant
to identify parameterisable values — hard-coded specifics that should
become dynamic inputs so the resulting skill is reusable.

Conversation:
{{ conversation }}

Skill name: {{ name }}
Skill description: {{ description }}

Find EVERY value in the conversation that is specific to this
particular invocation and would need to change when reused.

Examples of parameterisable values:
  - File paths and names     ("sales_q1_2025.csv" → input_file)
  - Dates and time ranges    ("2025-01-01"        → start_date)
  - Person/company names     ("Acme Corp"         → company_name)
  - Numeric thresholds       ("100"               → max_count)
  - URLs and endpoints
  - Column names, field names, or keys specific to the data

Do NOT parameterise:
  - Library names or tool choices (implementation decisions)
  - Output format choices integral to the skill's purpose
  - Values always the same for this type of task

For each parameter:
  - name:          snake_case identifier
  - type:          "string" | "integer" | "float" | "boolean"
  - description:   one sentence
  - example_value: EXACT value from the conversation, preserving
                   original spelling, casing, and whitespace
                   (if the column is "department", do NOT write
                   "Department")

The "preserve exact casing" rule

Generated skills would read a user-supplied CSV whose headers are in lowercase (eg 'department') but the script would reference row["Department"]. The model's training bias pushes it to "correct" casing; the runtime then explodes with KeyError. We patched this with an explicit "preserve EXACT spelling, casing, and whitespace" rule in the expand prompt.

backend/app/prompts/expand.md step 2 prompt · excerpt 
You are a Staff Engineer writing a skill package for Gemini CLI.
Use the CONVERSATION as ground truth for what the skill must do.

The primary file MUST be named 'SKILL.md' and MUST contain this
YAML frontmatter:

---
name:        {{ name }}
description: {{ description }}
category:    <one of: data, files, web, code, report, other>
---

The following parameters have been extracted from the conversation.
Replace ALL occurrences of the example values with the corresponding
{{param_name}} notation throughout SKILL.md and any scripts.

Extracted parameters:
{{ parameters }}

CRITICAL: Be highly opinionated and deterministic.
Do NOT offer multiple options or branching execution paths ("use
pandas OR csv"). Pick the single most robust approach.

CRITICAL: Preserve the EXACT spelling, casing, and whitespace of
all column names, field names, dictionary keys, and identifiers.
Do NOT "correct", capitalise, or normalise them. If the CSV header
is "department", use "department" — never "Department". Mismatched
casing causes runtime KeyError / column-not-found failures.

CRITICAL: Scripts must BOTH save their output AND print to stdout.
1. Save to `./output/` (bind-mounted — files persist and surface in
   the Files sidebar).
2. Also print a meaningful summary to stdout so the parent
   orchestrator can relay it to the user in the chat stream.

{{ additional_instructions }}

The validation ladder

Two softenings in Layer 2 matter more than they look.

  • Argparse unrecognized-args retry. If a script exits with unrecognized arguments, we strip the offending flags and retry once. Gemini sometimes hallucinates an --output flag when the script prints to stdout; this distinguishes "the script is broken" from "the caller passed a flag the script doesn't take."
  • Empty-stderr soft-pass. If a script exits non-zero with empty stderr, we treat it as a warning and move on. Most commonly the script needs a real input file and the read-only mount hit a file the user never uploaded. Treating this as an error would reject valid skills.

Auto-retry with error feedback

If validation fails, the preview() orchestrator loops up to three times. On each failure the concatenated error messages are injected back into the expand prompt as a "previous generation had errors" appendix, and Gemini regenerates.

backend/app/architect/preview.py python · excerpt 
async def preview(conversation: str, extra: str | None = None) -> SkillManifest:
    # Step 1: extract parameters (schema-constrained, temp=0.2)
    params = await extract_parameters(conversation)

    errors: list[str] = []
    for attempt in range(3):
        # Step 2: expand into a full skill, feeding prior errors back in
        manifest = await expand_skill(
            conversation, params,
            additional_instructions=extra,
            previous_errors=errors,   # empty on first attempt
        )
        report = await validate(manifest)   # AST + Docker dry-run
        if report.ok:
            return manifest
        errors = report.messages
        logger.info("retry %d: %d errors", attempt + 1, len(errors))

    # Surface the best-effort manifest + errors to the review modal.
    return manifest.with_warnings(errors)

Steering with use-case hints from the conversation

After validation, the skill surfaces in a review modal with an optional "any extra instructions?" box. To show the user what kind of refinement is useful, we pull concrete details from the conversation (column names, filter values, thresholds) and use them to build a context-aware placeholder — e.g., "filter status = active, sort by created_at descending" — so the empty box becomes a prompt that suggests how they can refine the skill.

Skill evolution

The same preview() function powers skill evolution. When a user refines an existing skill in natural language, we concatenate the instructions under an Additional instructions: delimiter; the extract step strips the suffix so it does not pollute parameter detection, and the expand step injects it via the {{ additional_instructions }} slot so the instructions shape both the body and the scripts.

Integrator — saving, mounting, versioning

The Integrator is the persistence layer. By the time the user clicks save, generation and validation have already happened — the Integrator's job is to write the approved skill to disk and mount it into future sessions.

On-disk layout

Skills live under backend/users/<user_id>/skills/<slug>/. Each folder follows the Gemini CLI convention: SKILL.md at root, plus optional scripts/, references/, assets/. No LLM call happens at save time — generation already happened during preview — so approval is instant.

Version snapshots on every mutation

Every update path snapshots the current state before overwriting, to a numbered sibling under versions/. Listing and restore are exposed via GET /api/skills/{slug}/versions and POST /api/skills/{slug}/versions/{v}/restore. Snapshot-then-overwrite order is load-bearing: if we overwrote first, a user evolving a skill and hating the result would have nothing to roll back to.

Mounting into the agent

At the start of every CLI call, _setup_session_workspace creates (or refreshes) a per-session working directory, copies the skill folders into .gemini/skills/, and seeds .gemini/ from a template. Active vs inactive is expressed as a .inactive marker file inside the skill folder. For long-running sessions, _sync_session_skills does a lazy diff and only adds or removes the delta — in most turns the diff is empty.

backend/app/integrator/session.py python · excerpt 
def _sync_session_skills(session_dir: Path, user_id: str) -> None:
    # Active skills = on disk under users/<uid>/skills/ without a .inactive marker.
    wanted = {s.name for s in active_skills(user_id)}
    mounted_dir = session_dir / ".gemini" / "skills"
    present = {p.name for p in mounted_dir.iterdir()} if mounted_dir.exists() else set()

    to_add = wanted - present
    to_remove = present - wanted
    if not to_add and not to_remove:
        return   # the hot path — empty diff on most turns

    for slug in to_add:
        shutil.copytree(skill_path(user_id, slug), mounted_dir / slug)
    for slug in to_remove:
        shutil.rmtree(mounted_dir / slug)

From the user's perspective, none of this plumbing is visible — they just open a new chat and use the skill.

The Docker sandbox

Everything the agent executes runs inside a per-session Docker container.

One container per session, long-lived

On the first message, the backend spawns a container autoskill-session-{session_id} from the autoskill-sandbox image, running tail -f /dev/null. Every subsequent message in the session runs through the same container via docker exec, so conversational state (Gemini CLI --resume), cached dependencies, and in-workspace files all persist at near-zero overhead. A background task removes the container and its workspace once a session has been silent for 2 minutes.

The mount layout

The agent can write to its working directory freely — that is how it produces artifacts.

Per-user images

Skills can declare a requirements.txt. When a user's library changes, we aggregate all requirements.txt files into a master list and build a personalised image layered on top of autoskill-sandbox in the background. If the user opens a chat before the build finishes, that first session falls back to the base image and installs dependencies on startup; once the personalised image is ready, every subsequent session starts with pip install already done.

Validation sandboxes are separate

The validation sandbox is distinct from the chat sandbox. Each script dry-run spins up a fresh docker run --rm, not an exec into the live session. This matters for isolation — a half-validated script from a candidate the user hasn't approved yet should not influence the session's live workspace.

Read-only skill mount

The .gemini/skills directory is mounted :ro into the session container. The agent can use skills but not edit them: any change has to go back through the review modal, which re-runs the full validation ladder. Trust boundaries belong at the layer that produces new code, not the layer that runs it — if the agent could rewrite its own skills at runtime, the Architect's validation would be cosmetic.

Beyond the proposal

Four features grew out of using the system ourselves.

Duplicate-skill detection

The first time a teammate ran the same workflow twice with slightly different wording, the system happily offered a second copy. We added an embedding-based check: the candidate's name + description and every saved skill's embed together in a single gemini-embedding-001 batch call; cosine similarity above 0.6 attaches a SimilarSkillMatch to the candidate. The frontend surfaces it as an amber warning with an "Update Existing" button.

Skill evolution

When the duplicate detector fires, the review modal opens in a side-by-side comparison (see Figure 6). Submitting refines the existing skill rather than creating a new one, via POST /api/skills/{slug}/evolve. The previous version is snapshotted automatically; one click restores it.

Worker profiles

A user with ten skills doesn't always want all ten available. Workers are named bundles — "Data Ops," "Report Writer" — that override active/inactive flags for the session they're launched under. When a chat session is launched with a worker_id, the workspace-sync logic restricts the mounted skills to exactly the worker's list.

Session-scoped Files panel

Any file the agent generates inside /app/output is persisted and surfaced in a right-hand Files sidebar, streamed to the frontend via an SSE event after each chat turn. Files have preview, one-click download, humanised sizes and dates, and delete. They are session-scoped by design — the panel clears on "New Chat" or when the zombie cleaner reaps the session.

Skill usage tracking

Every successful invocation of a skill is recorded to a per-user usage.json. The "successful" qualifier is load-bearing: the tool-result handler scans output for error signatures (Traceback, No such file, Permission denied, Error:) and only credits the skill if none are present. Counts feed the Skills sidebar's sort order, so the most-used skills float to the top — and a skill that errors out doesn't inflate its way up the list.

The frontend surface

Next.js 16 + React 19 in TypeScript, with Tailwind for layout and a small set of single-purpose components. We kept the surface deliberately flat — one chat pane, two collapsible sidebars (Skills left, Files right), a top bar.

ComponentRole
SkillsSidebar.tsxLibrary with category-coloured pills, pending candidates, similar-skill warnings, History tab.
SkillReviewModal.tsxCreate mode (single-panel preview, Monaco editor) and Evolve mode (side-by-side with NL refinement).

Evaluation

Status: the evaluation module (evaluation/) has not been fully restored to the submitted repo at time of writing. Only the compiled .pyc files from earlier runs remain; the source scenarios and the runner.py entry point are missing. The numbers below are the design targets we set during the proposal; actuals will replace them once the module is reinstated and a fresh run is captured. We are flagging this rather than reporting fabricated results.

Evaluation was designed to run offline via evaluation/runner.py, independently of the frontend. Two suites, both fed a fixed set of hand-authored YAML scenarios: one scores trigger detection, the other runs the full Architect pipeline twice and compares the outputs.

Quantitative — trigger detection

The trigger suite feeds the evaluator YAML-defined scenarios through evaluate_trigger() and scores three rates. Positive cases are CSV/data workflows, multi-step API calls, and file-generation tasks. Negative cases are the ones the trigger has to actively avoid: "write me a plan," single-question lookups, chit-chat, and turns where an existing skill already handled the work.

Qualitative — consistency and completion

The consistency suite runs the full Architect pipeline on the same scenario twice and compares: do both produce a valid, sandbox-executable skill (completion), do they share the same script files and section headings (structural), and is the extracted parameter set the same across runs (parameter). Because the pipeline is non-deterministic by design, this is the only signal we trust for "does the system produce the same answer twice."

Consistency matters more than raw detection: a pipeline that triggers reliably but generates subtly different skills on the same input is unusable. Until the module is restored, we are deliberately not reporting numbers we cannot stand behind.

Deployment

Three moving parts: frontend, backend, and the Docker daemon the backend talks to.

frontend
Next.js on Vercel. One env var: NEXT_PUBLIC_BACKEND_URL.
backend
EC2 behind a DuckDNS domain (auto-skill.duckdns.org) with a Let's Encrypt cert terminating TLS at port 443. The host has Docker installed and docker.sock mounted into the backend container, so FastAPI can docker run session containers on the host — a DinD-style setup. Cert auto-renews via the standard certbot cron.
Gemini billing
Calls are quota-capped on the Gemini API; we stayed inside the free tier for normal demo usage. Flash Lite for the trigger (cheap, high volume), the main model for extract/expand (rarer, heavier).
cost
EC2 is the main fixed cost (~$1/day on-demand). Gemini calls are the variable cost.

Limitations

File-based persistence

Skills and workers persist to disk under backend/users/<user_id>/ and survive sessions, restarts, and redeploys. The session buffer and usage counts live in process memory and are lost on restart. The file-based store was a deliberate scope choice for the proof-of-concept — zero infra, fast iteration — but it caps real-world scale at one backend process. A production deployment would swap in Postgres for skill metadata and Redis for the buffer.

Cross-session workflows

The Observer only sees the current session's buffer, so a workflow that spans two sessions ("yesterday you wrote this script; today, update it") does not trigger as a single unit. Arguably a feature rather than a bug, because it forces the workflow to be stably reproducible in one sitting before it is saved. A cross-session trigger would also need to disambiguate "continuing yesterday's work" from "starting something new that resembles yesterday's work."

Session-scoped generated files

Files the agent generates are tied to the active session — when the session ends, cleanup_session() wipes the workspace and the generated files with it. Skills and workers are unaffected; only session artifacts go. Most generated files are ephemeral by nature (a filtered CSV, a one-off summary PDF), so this is the right default; the friction point is users occasionally producing a file they didn't realise they wanted to keep until after the session closed. A per-file "keep this" toggle that promotes a generated file into the durable library would be the natural next step.
